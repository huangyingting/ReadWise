## 2026-06-18 11:25 UTC - US-026 Background processing worker

  `listUnprocessedArticleIds({limit:batchSize})` and runs each id through the idempotent

  Failures = thrown error OR `result.ok===false`; after `maxRetries` it gives up (counts failed,
  never blocks the queue). Resumes pending work automatically on restart (queue is the source of
  truth; processor is cache-first — no checkpoint state). Returns `WorkerStats`.
- Safe stop via `AbortSignal`: interruptible `sleep(ms, signal)` rejects with AbortError so idle/
  backoff waits stop promptly; the loop swallows AbortError and resolves cleanly. `--once` drains
  the queue then exits. `createConsoleLogger()` = timestamped/level/JSON-meta lines (placeholder
  until US-029). All deps injectable via `opts.deps` for DB-free unit tests.
- Added CLI `scripts/worker.ts` (`npm run worker -- ...`, reuses the scraper TS-CLI harness):
  flags `--once`, `--interval <ms>`, `--batch <n>`, `--max-retries <n>`, `--backoff <ms>`,
  `--include-published`, `--tts`, `--translate <codes>`, `--help`. Wires SIGINT/SIGTERM →
  controller.abort() (2nd signal force-exits 130). Warns (not errors) when AI/Speech unconfigured.
- Files: src/lib/worker.ts (new), scripts/worker.ts (new), package.json (+worker script), AGENTS.md.
- Quality: typecheck PASS, lint PASS (0 warnings).

  it (difficulty/tags/vocab/quiz generated), published draft→published, then drained+stopped. Continuous

  (retried=1, published=1); exhausted retries counted failed with exactly 1+maxRetries calls. Cleaned up

- **Learnings for future iterations:**
  - `unstable_cache` serializes the cached result, so Date columns come back as ISO STRINGS on a
    cache HIT. Any `.getTime()`/date math must happen INSIDE the cached fn (e.g. listRelatedArticles
    sorts before returning). All current consumers go through `toListingArticle` (date-free), so this
    is safe — but watch it when adding new consumers that read article dates.
  - `revalidateTag` only works inside a Next request scope (Route Handler / Server Action). The CLI
    worker/processor runs outside that, so guard the call (try/catch) and rely on the time-based
    `revalidate` window for out-of-band publishes.
  - `unstable_cache` automatically folds the wrapped function's arguments into the cache key, so
    pass distinguishing values (category, offset, limit, level, topics) as plain positional args
    rather than stuffing them into `keyParts`.
  - Verify caching end-to-end by mutating the DB DIRECTLY (bypassing the API) to prove a stale read,
    then mutating via the API to prove invalidation — a clean before/after without needing AI config.
  - `next start` got into an unresponsive state mid-test once; `next dev` was reliable for the
    runtime cache verification.
