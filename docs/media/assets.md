# Media asset lifecycle

This document covers `MediaAsset` ownership: how assets are created, keyed,
tracked, served, migrated, and deleted. Storage backend configuration and
migration procedures are in [`storage.md`](./storage.md).

## Ownership boundary

**Media subsystem owns** the `MediaAsset` table, storage keys, content checksums,
MIME types, byte sizes, duration metadata, storage migration and rollback, asset
deletion, and orphan handling.

**Media does not own** reader playback UX, TTS job scheduling, or the TTS
provider. See [`../reader/playback.md`](../reader/playback.md) and
[`../speech/generation.md`](../speech/generation.md) for those boundaries.
[`../operations/tts-jobs.md`](../operations/tts-jobs.md) covers scheduling.

## MediaAsset model

`MediaAsset` records metadata for every object stored outside the primary
database. Speech narration is the only asset kind currently in use (`kind =
"speech"`).

| Field        | Type     | Description                                              |
| ------------ | -------- | -------------------------------------------------------- |
| `id`         | String   | CUID primary key.                                        |
| `storageKey` | String   | Unique, content-addressed object key (`speech/<sha256><ext>`). |
| `kind`       | String   | Asset type â€” currently always `"speech"`.               |
| `mimeType`   | String   | MIME type of the stored bytes (e.g. `audio/mpeg`).      |
| `sizeBytes`  | Int      | Byte length of the stored object.                        |
| `checksum`   | String   | Lowercase hex SHA-256 of the stored bytes.              |
| `durationSec`| Float?   | Audio duration in seconds (from word-boundary timings). |
| `voice`      | String?  | Azure Speech voice name used during synthesis.          |
| `format`     | String?  | Azure Speech output-format string.                      |
| `articleId`  | String?  | Article that owns this asset. Cascade-deleted with it.  |

The `MediaAsset` row is created (upserted) by
`src/lib/speech/repository.ts:saveSpeechResult` at synthesis time, and by
`src/lib/storage/speech-migration.ts:migrateArticleSpeechToStorage` during
bulk migration.

## Storage keys

Keys are content-addressed and generated by `src/lib/storage/key.ts`:

```
speech/<sha256hex><ext>
```

- `sha256hex` is the lowercase hex SHA-256 of the raw audio bytes.
- `ext` is inferred from the MIME type (`.mp3`, `.ogg`, `.wav`, `.webm`).
- Identical audio bytes always produce the same key â€” writes are idempotent.
- Keys are traversal-safe: the filesystem backend confines all keys to the
  configured base directory; the Azure backend uses SDK container scoping.
- Storage keys are **never exposed** directly to client code or API responses.

## Checksums and deduplication

The `checksum` field (hex SHA-256) serves two purposes:

1. **Integrity verification** â€” callers can verify the retrieved bytes match the
   recorded checksum.
2. **Deduplication** â€” because the key embeds the checksum, re-synthesizing
   identical text with the same voice and format produces the same key, and the
   `MediaAsset` upsert (`where: { storageKey }`) avoids duplicate rows.

## Storage migration

`migrateArticleSpeechToStorage()` (`src/lib/storage/speech-migration.ts`) lifts
existing inline base64 audio into object storage. See [`storage.md`](./storage.md)
for the full migration procedure including rollback instructions.

**Idempotency predicate:** only rows where `audioBase64 IS NOT NULL` AND
`storageKey IS NULL` are eligible. Re-running the migration processes nothing
new.

**Safety guarantee:** `audioBase64` is cleared only after both the storage write
and the `MediaAsset` upsert succeed inside a transaction. The payload is never
lost.

## Article deletion and cascade

`MediaAsset` has `onDelete: Cascade` on the `articleId` foreign key. Deleting an
article removes its `MediaAsset` rows from the database automatically.

**The storage object is not automatically deleted when the database row is
removed.** Operators who want to reclaim storage space must run a separate
cleanup pass â€” iterate `MediaAsset` rows with `articleId IS NULL` (orphaned
because the article cascade fired) and call `storage.delete(storageKey)` for
each.

## AI rebuild and asset replacement

`adminRebuildArticleAI(id, ...)` in `src/lib/article-library/admin.ts`:

1. Deletes the `ArticleSpeech` row for the article.
2. Calls `mediaAsset.deleteMany({ where: { articleId: id, kind: "speech" } })` to
   remove the `MediaAsset` database record.
3. Does **not** delete the storage object â€” the object becomes unreferenced in the
   database but the bytes remain in storage until a cleanup pass removes them.

After a rebuild, the next reader request or `TTS_GENERATE` background job
re-synthesizes narration and creates a new `MediaAsset` with fresh storage key,
checksum, and metadata.

## Orphan handling

An orphan is a storage object whose `MediaAsset` row has been deleted (via
article cascade or admin rebuild) but whose bytes remain in object storage.

Causes:
- **Article deletion** â€” cascade removes `MediaAsset` but not the storage object.
- **Admin AI rebuild** â€” `MediaAsset` is deleted explicitly; object is retained.
- **Failed migration cleanup** â€” partially migrated rows where the DB transaction
  succeeded but a subsequent manual deletion removed the row.

Detection: query `MediaAsset` for rows with `articleId IS NULL` or cross-check
storage keys in `ArticleSpeech.storageKey` against `MediaAsset.storageKey`.

Cleanup: call `getMediaStorage().delete(storageKey)` for each orphaned key, then
delete the `MediaAsset` row if it still exists.

**Do not delete the storage container** until you are certain every row has
either been migrated (storageKey set) or has base64 as a fallback.
