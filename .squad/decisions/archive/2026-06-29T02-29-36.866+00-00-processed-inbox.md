## What shipped

A low-pressure post-reading **comprehension self-check** that completes the Today
comprehension step **without a full article quiz**, plus a gentle **remediation**
loop, feeding structured weakness signals into the **existing** mastery system.

### Flow
1. After `readingCompletedAt`, the Today page offers an optional self-check:
   a single self-rating (`confident`/`partial`/`confused`) + **zero or one**
   lightweight MCQ drawn from the article's existing `QuizQuestion` rows
   (most recently added; tag-agnostic since the schema carries no per-question tag).
2. **Self-rating alone advances `comprehensionCompletedAt`** — no forced quiz.
3. **Wrong MCQ answer** → remediation card with "Go back to the article" deep-link
   (non-AI; `remediationViewed` flag persisted).
4. Weakness signals (self-rating + MCQ outcome via `skillTag`) feed the EXISTING
   `updateArticleMastery` + `recordSkillEvidence` paths — best-effort; a mastery
   failure never breaks completion.

### Files
- NEW `src/lib/engagement/today-session/comprehension.ts` — controlled
  self-rating/skill-tag constants, MCQ selection (never leaks `correctIndex`),
  `submitTodayComprehension` orchestration (server-side grading, idempotent
  `TodayComprehensionFeedback` upsert, mastery, remediation, analytics).
- NEW `GET/POST /api/today/comprehension` — auth + `isTodaySessionFeatureEnabled()`
  flag-guard (404 off) + validation; userId never read from body.
- NEW `TodayComprehensionCheck` UI (design-system primitives + tokens), wired into
  the Today page above `TodayWorkflow`.
- EDIT `completion.ts`: `markTodayComprehensionComplete` accepts optional `selfRating`.
- EDIT `analytics.ts`/`catalog.ts`: new `today_comprehension_submitted` event.
- EDIT `account-commands.ts`: `exportUserData` includes `TodayComprehensionFeedback`
  (controlled fields only).
- EDIT `index.ts` barrel exports; regenerated API catalog.

## Mastery signal wiring
- `comprehension` skill ← self-rating (confident 0.9 / partial 0.6 / confused 0.3,
  weight 0.5 — low, informational).
- Mapped skill ← MCQ outcome (1/0): `main_idea`/`detail`/`inference` → `comprehension`;
  `vocabulary_in_context` → `vocabulary`.
- `updateArticleMastery` refreshes `ArticleMastery.comprehensionScore`.
- Reuses existing APIs only — no parallel mastery path. Did NOT touch
  study-plan-engine / recommendations / coach-memory (sibling-owned).

## Privacy
Persisted row + analytics payload carry **ids / enums / booleans ONLY**. MCQ graded
**server-side** (`selectedIndex` → cached `correctIndex`) so the answer is never sent
to the client. Dedicated privacy tests assert the DB row and the analytics payload
contain no question/option/article text.

## Deviations (honest)
- Design referenced `QuizQuestion.explanation` and a per-question `skillTag`, but the
  merged schema's `QuizQuestion` has neither. Phase-1 remediation therefore degrades
  to a generic low-pressure message + article deep-link (no stored explanation to

- Chose server-side grading (`selectedIndex`) over the design's body `mcqCorrect`
  for integrity + to avoid leaking the answer to the client.
- Surfaced the MCQ via a dedicated `GET /api/today/comprehension` rather than polluting
  the privacy-safe Today view-model with question text.

# Decision: Content-based HTTP-200 bot-challenge detection in fetch chain

- **Author:** Livingston (Backend Dev)
- **Date:** 2026-06-28
- **PR:** #846 (MERGED to main, squash, branch deleted)
- **Files:** `src/lib/scraper/fetch-strategies.ts`, `tests/scraper-fetch-strategies.test.ts`

## Problem
The multi-strategy fetch chain escalated only on bot-challenge **status codes**
(401/403/429/451/503). Cloudflare / Vercel / DataDome return **HTTP 200 with an
interstitial challenge page**, so the chain accepted the interstitial and the
article extracted to null. Proven live: aeon → `<title>Vercel Security
Checkpoint</title>`; undark → `<title>Just a moment...</title>` (Cloudflare).

## Solution
Added exported `looksLikeBotChallenge(html, status?)`:
- **Signals (case-insensitive vendor markers):** Cloudflare (`Just a moment...`,
  `Attention Required! | Cloudflare`, `Checking your browser before accessing`,
  `cf-browser-verification`, `cf-challenge`, `Performing security verification`,
  `Enable JavaScript and cookies to continue`, `__cf_chl`), Vercel
  (`Vercel Security Checkpoint`, `We're verifying your browser`), DataDome /
  PerimeterX / Akamai (`DataDome`, `px-captcha`, `Access to this page has been
  denied`, `Pardon Our Interruption`).
- **Generic heuristic:** visible-text < ~250 chars + `noindex` robots meta + no
  article markers → challenge.
- **False-positive guard (runs first):** any body with `<article>`, ≥3 `<p>`,
  JSON-LD, or `og:title` is NEVER a challenge — protects short real articles.
- A bot-challenge **status** still short-circuits to true.

## Wiring
- After each strategy returns a 2xx body, run `looksLikeBotChallenge`. If true,
  treat as BLOCKED and escalate exactly like a 403 (next profile → reader →
  wayback). Applies to origin, profile-retry, AND the reader result.
- If the whole chain only ever yields challenge pages, throw `BotChallengeError`
  ("bot challenge not bypassed for <host>") so extraction never receives a
  challenge interstitial.
- Per-host memory unchanged (only remembers a strategy that returned real
  content, since challenge bodies never reach `hostStrategyMemory.set`).

## Security (preserved)
SSRF validation of the original URL first; reader/wayback only for validated
public URLs + fixed host allowlist (r.jina.ai, web.archive.org); robots posture
unchanged; reader header `X-Return-Format: html` unchanged; 404/410/SSRF/timeout
behavior unchanged.

## Tests
200 Cloudflare/Vercel challenge → escalates (later real strategy wins); real
(and SHORT real) 200 article → no escalation; all-challenge → throws blocked
error (does not return challenge HTML); `looksLikeBotChallenge` unit cases
(vendor true, real false, tiny+noindex true, status short-circuit). No real
network.

- `npm run typecheck` — clean
- targeted scraper-fetch* + ssrf — 36 pass
- `npm test` — 3249 pass / 0 fail / 22 skip
- eslint touched files — clean

## Note
aeon/undark need a `JINA_API_KEY` for the premium reader to actually SOLVE their
challenges. This detection enables that path. Without a key, those specific

# Decision: Multi-strategy scraper fetch fallback chain

- **Author:** Livingston (Backend Dev)
- **Date:** 2026-06-27T23:45Z
- **PR:** #845 — MERGED (squash, mergeCommit 7a55a7f). Branch `squad/scraper-fetch-strategies` deleted.
- **Requested by:** huangyingting

## What
Added a fallback chain to the SSRF-safe **GET** fetch (`fetchHtml`) so bot-blocked
provider pages (Cloudflare/DataDome 401/403/429/451/503) can still be captured.
`fetchText` (POST) unchanged. New module `src/lib/scraper/fetch-strategies.ts`
(`@server-only`), wired into `fetchHtml`. Inspired by the `fetch-url` skill but
implemented natively inside ReadWise's existing fetch — no external script vendored.

## Chain order (origin → profile-retry → reader → wayback)
1. **origin** — existing default request, unchanged for 2xx pages (backward compatible).
2. **browser-profile retry** — rotating realistic UA + header sets, order:
   `googlebot` (first, backward-compat), `desktop-chrome`, `desktop-firefox`,
   `desktop-safari`, `mobile-safari`, `bingbot`. Desktop profiles add
   `Accept`, `Accept-Language`, `Sec-Fetch-Mode: navigate`, `Sec-Fetch-Site: none`,
   `Sec-Fetch-Dest: document`, `Upgrade-Insecure-Requests: 1`. Bot/mobile profiles
   use Accept + Accept-Language only.
3. **reader** — `https://r.jina.ai/<originalUrl>`, header `X-Return-Format: html`,
   plus `Authorization: Bearer <JINA_API_KEY>` when the env key is set (works without it).
4. **wayback** — `https://web.archive.org/web/<YYYY>id_/<originalUrl>` (the `id_`
   suffix returns the original page without the Wayback toolbar; bare year resolves
   to the closest snapshot via redirect). Availability API intentionally skipped for
   simplicity/robustness.

If everything fails, the first bot-challenge error is rethrown (as today).

## Security preserved
- **SSRF**: original URL is `assertSafeUrl`-validated BEFORE any request, so
  reader/wayback URLs are only built from an already-validated public origin.
  Origin/profile requests run through `fetchCore` (re-validates + IP-pins every
  redirect hop). Reader/wayback target a FIXED host allowlist
  (`r.jina.ai`, `web.archive.org`), asserted before fetching.
- **Only bot-challenge statuses (401/403/429/451/503) advance the chain.** Genuine
  not-found (404/410), SSRF rejection, timeout, too-many-redirects, and network
  errors on an origin/profile attempt ABORT immediately — no fallback. Implemented
  via a new `FetchHttpError` (carries `.status`) thrown by `fetchCore`; message
  format `HTTP <status> for <url>` preserved for backward-compat.
- **robots** unchanged — enforced at discovery via `isUrlAllowed`; no robots bypass
  added in the fetch layer.
- Per-attempt timeout respected (`scraperTimeoutMs`); an overall budget
  (4× per-attempt) caps the whole chain so it can't hang.

## Per-host memory
Process-lifetime `Map<host, winningStrategy>` (no DB/file persistence). Once a host
is known to need e.g. reader or a specific profile, subsequent fetches try that
strategy FIRST and skip the slow chain. Resets on process restart.

## Env flags (runtime-config/scraper.ts + .env.example)
- `SCRAPER_FETCH_PROFILE_RETRY` (default **true**)
- `SCRAPER_FETCH_READER` (default **true**)
- `SCRAPER_FETCH_WAYBACK` (default **true**)
- `JINA_API_KEY` (optional)

Each follows the existing `!== "false"` default-ON pattern (like `scraperReadability`).

## Tests
New `tests/scraper-fetch-strategies.test.ts` (11 tests, network fully mocked via
`mock.module` of `@/lib/scraper/ssrf` + `undici`, matching existing test style).
Asserts: profile retry order (googlebot 2nd, chrome wins w/ sec-fetch headers),

# Decision: Clean-capture pipeline integration into the scraper

**Author:** Rusty (Lead/Architect)
**Date:** 2026-06-27
**PR:** #841 (squash-merged into main)
**Status:** MERGED

## Summary
Rewired `src/lib/scraper/extract.ts` `extractArticle()` body step to use the
merged `extractReadable` (linkedom + @mozilla/readability) and
`declutterArticleHtml` modules, with a content-preserving fallback and a new
`SCRAPER_READABILITY` kill-switch (default ON).

## New body pipeline
1. `legacyBody` = JSON-LD `articleBody` (via `paragraphsToHtml`) when present,
   else the raw `<p>` harvest (`extractBodyHtml`).
2. When `scraperReadability()` is true, `readable = extractReadable(cleanedHtml, sourceUrl)`
   — uses the **cleaned, NOT normalized** HTML so Readability sees full structure.
3. Body choice (never lose content):
   - **JSON-LD `articleBody` is canonical structured text → always kept.**
     Readability never overrides it.
   - For the raw-`<p>` path, Readability wins **unless** the legacy body is
     `>1.5×` longer (constant `READABILITY_LEGACY_MAX_WORD_RATIO = 1.5`),
     which signals Readability over-trimmed → fall back to the longer legacy body.
4. `declutterArticleHtml(chosenBody, { byline: readable?.byline ?? author, authorName: author })`
   runs in BOTH paths — removes residual boilerplate, especially the trailing
   author byline/bio. Conservative 35% removal guard.
5. `sanitizeArticleHtml(decluttered).trim()` — unchanged final security pass.
6. 50-word floor unchanged.
7. Metadata fallbacks only: `author ?? readable.byline`,
   `description/excerpt ?? readable.excerpt`. Existing JSON-LD/OG values never
   overridden. `ScrapedArticle` has no `lang` field → lang not surfaced.

## Key decision / deviation
**Readability does not override a JSON-LD `articleBody`.** The mission's literal
heuristic (pure word-count compare) would have let the page's raw `<p>` harvest
win over JSON-LD in the "JSON-LD overrides raw <p>" test. JSON-LD `articleBody`
is the cleanest canonical body (no trailing bios — that residue only appears in
the DOM-harvest path), so gating Readability on `!ldBody` is the correct
correctness call and preserves that meaningful existing assertion. Documented
in code comments.

Because of this design, **no existing scraper/provider test assertion needed to
be changed or weakened** — Readability preserved paragraphs/content on all
raw-harvest fixtures, and JSON-LD fixtures kept their canonical bodies.

## Kill-switch
`SCRAPER_READABILITY` (env, default TRUE), style mirrors `scraperHtmlNormalize()`
in `src/lib/runtime-config/scraper.ts`. `SCRAPER_READABILITY=false` ≈ today's
pipeline PLUS the always-safe declutter pass.
